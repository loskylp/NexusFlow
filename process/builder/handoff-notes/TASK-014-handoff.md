**Date:** 2026-03-29
**Task:** Pipeline chain definition
**Requirement(s):** REQ-014
**ADRs:** ADR-003 (idempotency), ADR-008 (data model)

## What Was Implemented

### New files

**`internal/db/migrations/000004_chains.up.sql`**
Adds two normalised tables:
- `chains` — top-level chain record (id UUID PK, name TEXT, user_id UUID FK, created_at TIMESTAMPTZ)
- `chain_steps` — one row per pipeline step (chain_id FK, pipeline_id FK, position INT, PK(chain_id, pipeline_id), UNIQUE(chain_id, position))

`UNIQUE(chain_id, position)` enforces no two steps at the same position (no branching at the DB layer). `UNIQUE(chain_id, pipeline_id)` enforced by the composite PK enforces no duplicate pipelines in a chain.

**`internal/db/migrations/000004_chains.down.sql`**
Drops `chain_steps` then `chains`.

**`internal/db/queries/chains.sql`**
Six sqlc queries: `CreateChain`, `GetChainByID`, `CreateChainStep`, `GetChainSteps`, `FindChainByPipeline`, `GetNextPipelineInChain`. Regenerated via `sqlc generate` (Docker, sqlc:latest). Generated output: `internal/db/sqlc/chains.sql.go` (new), `internal/db/sqlc/models.go` (extended with `Chain` and `ChainStep` structs).

**`internal/db/chain_repository.go`**
`PgChainRepository` implementing `db.ChainRepository`:
- `Create` — transactional: inserts chain header + all steps atomically.
- `GetByID` — fetches chain header then assembles PipelineIDs from GetChainSteps rows.
- `FindByPipeline` — used by chain trigger; finds the chain containing a given pipeline.
- `GetNextPipeline` — returns the next pipeline UUID in the chain, nil if last step.

**`api/handlers_chains.go`**
`ChainHandler` with two endpoints:
- `Create` (POST /api/chains) — validates name is non-empty, at least two pipeline IDs, no duplicates (duplicate = branching structure, rejected 400). Returns 201.
- `Get` (GET /api/chains/{id}) — returns chain with ordered PipelineIDs. Returns 404 on missing.

Private helpers: `parseAndValidatePipelineIDs` (parse + deduplicate-check), `parseChainID` (chi UUID extractor).

**`api/handlers_chains_test.go`**
8 unit tests covering: 201 linear chain, 400 branching (duplicate pipeline IDs), 400 empty pipeline list, 400 single pipeline, 401 unauthenticated, 200 get with ordering, 404 missing chain, 400 invalid UUID. Uses `stubChainRepo` (in-memory `db.ChainRepository`) and `newCapturingPipelineRepo` from TASK-013 tests.

**`worker/chain_trigger.go`**
`ChainTrigger` struct with:
- `OnTaskCompleted(ctx, taskID, pipelineID, userID)` — the hook called after task completes. Checks if pipelineID is in a chain, finds the next pipeline, acquires a SET-NX idempotency key `"chain-trigger:{taskID}:{nextPipelineID}"`, then submits the downstream task.
- `chainTriggerKey(taskID, nextPipelineID)` — builds the idempotency key string.

`ChainTaskEnqueuer` interface — isolates submission from trigger logic.
`ChainIdempotencyStore` interface — satisfies the SET-NX contract; `queue.RedisQueue` implements it.

**`worker/chain_enqueuer.go`**
`WorkerChainEnqueuer` implementing `ChainTaskEnqueuer`:
- `SubmitChainTask` — creates a Task in PostgreSQL (status "submitted"), enqueues it on Redis Streams, then advances to "queued" via `UpdateStatus`. Mirrors the API task submission flow.

**`worker/chain_trigger_test.go`**
4 unit tests covering: submits next pipeline on completion, idempotency (duplicate completion event submits only one downstream task), no-op for last pipeline in chain, no-op when pipeline is not in any chain. Uses in-memory stubs for all dependencies; no Redis or PostgreSQL required.

## Design Decisions

**Normalised chains table (not UUID array).** The task specified new `chains` and `chain_steps` tables. The existing `pipeline_chains` UUID-array table (migration 000001) is retained — it has a FK from `tasks.chain_id` and cannot be dropped without altering the tasks schema. The new normalised tables are what TASK-014 uses; `pipeline_chains` is legacy.

**`tasks.chain_id` not set by chain trigger.** The `tasks.chain_id` column FKs to `pipeline_chains` (old table), not to `chains` (new table). Chain-triggered tasks have `chain_id = nil`. This is correct within the TASK-014 scope — the tasks.chain_id FK to `pipeline_chains` is a TASK-002/TASK-005 artifact; updating it to reference `chains` would require a schema migration not in scope here.

**Idempotency key format.** `"chain-trigger:{taskID}:{nextPipelineID}"` uniquely identifies a chain trigger event. Two components are needed because the same task can complete multiple times (redelivery) but should produce only one downstream task; the nextPipelineID anchors the key to the specific trigger.

**`WithChainTrigger` over constructor argument.** Adding `chainTrigger` to the constructors would break every test that constructs a Worker. The fluent setter keeps all existing call sites unchanged and makes the chain trigger opt-in.

## Known Limitations

1. **tasks.chain_id not set on chain-triggered tasks.** Chain-triggered tasks have `chain_id = nil` because the FK references `pipeline_chains`, not the new `chains` table. A future migration (e.g. 000005) could add a `chain_ref_id` column referencing `chains.id` for richer tracking, or migrate the FK to point to `chains`.

2. **Chain task tags.** `WorkerChainEnqueuer` uses `cfg.WorkerTags` (the worker's own tags) to route the downstream task. If the next pipeline requires a different worker tag, the task will be routed to an incompatible worker. Tag routing per chain step is a future enhancement not in scope for TASK-014.

3. **No GET /api/chains (list all).** Only `GET /api/chains/{id}` is implemented. The acceptance criteria do not require a list endpoint; it can be added in a future task.
